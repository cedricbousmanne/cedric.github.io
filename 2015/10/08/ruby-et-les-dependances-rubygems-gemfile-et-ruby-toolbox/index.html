Ruby et les dépendances : RubyGems, Gemfile et Ruby Toolbox

Oct 8, 2015 • cedric

Quand on parle de Ruby on Rails, RubyGems n’est jamais très loin. RubyGems, c’est le service d’hébergement des Gems, l’écosystème des bibliothèques (librairies) inhérantes à Ruby.

A ce jour, RubyGems héberge plus de 6000 Gems et totalise plus de 6 milliards de téléchargements. Ce n’est pas étonnant, mais ça n’en est pas moins impressionnant.

Anatomie d’un Gemfile

Le fichier Gemfile, c’est un peu la liste de courses de votre application. Vous pouvez y préciser l’ensemble des librairies requises pour faire tourner votre app.

Ce fichier doit obligatoirement être installé à la racine de votre projet. Vous pouvez le constater dans l’arborescence ci-dessous, tirée d’un projet Rails tout neuf.

Source

Sur la première ligne, source permet de définir le(s) dépôt(s) à consulter pour aller y télécharger les gems précisées plus bas dans le fichier.

Par défaut, seul le dépot de rubygems.org est défini, mais on peut en ajouter d’autres.

Gemfury, par exemple, est un service d’hébergement de gems privé. Une fois inscrit au service, vous devrez ajouter une ligne source au Gemfile de votre application pour pouvoir y télécharger les gems que vous aurez hébergées dessus.

Les lignes gem

Les lignes gem permettent de spécifier les gems à installer pour faire tourner votre application. C’est une liste de dépendances, ni plus ni moins.

Vous aurez remarqué que certaines lignes prenaient un second paramètre, il s’agit du numéro de version de la gem que l’on souhaite utiliser.

Ce même numéro de gem est parfois préfixé d’autres caractères, tels que ~> ou >=, il s’agit là d’une indication autorisant une certaine souplesse dans le numéro de version requis.

<span class="c1"># (...)</span>
<span class="n">gem</span> <span class="s1">'rails'</span><span class="p">,</span> <span class="s1">'4.2.0'</span>
<span class="n">gem</span> <span class="s1">'sqlite3'</span>
<span class="n">gem</span> <span class="s1">'sass-rails'</span><span class="p">,</span> <span class="s1">'~> 5.0'</span>
<span class="n">gem</span> <span class="s1">'uglifier'</span><span class="p">,</span> <span class="s1">'>= 1.3.0'</span>
<span class="n">gem</span> <span class="s1">'coffee-rails'</span><span class="p">,</span> <span class="s1">'~> 4.1.0'</span>
<span class="c1"># (...)</span>
Explications
  • gem 'sqlite3' : Aucun numéro de version n’est précisé, la dernière version sera automatiquement installée
  • gem 'rails', '4.2.0' : La version 4.2.0 sera utilisée. Aucune autre.
  • gem 'coffee-rails', '~> 4.1.0' : La version 4.1.0 sera utilisée, en étant permisif sur le dernier digit. La version 4.1.1, si elle existe, pourra être prise en compte. En revanche, la version 4.2.0 ne sera pas autorisée.
  • gem 'sass-rails', '~> 5.0' : Comme pour la ligne précédente, les version 5.0, 5.1, etc. seront acceptées.
  • gem 'uglifier', '>= 1.3.0' : La version la plus récente, pourvu qu’elle soit supérieure ou égale à 1.3.0

Les blocs group

Les blocs group permettent d’isoler certaines gems en fonction de l’environnement voulus. Dans l’exemple ci-dessous, les gems byebug, webconsole et spring ne seront accessible que dans les environnements development et test.

<span class="c1"># (...)</span>
<span class="n">group</span> <span class="ss">:development</span><span class="p">,</span> <span class="ss">:test</span> <span class="k">do</span>
  <span class="n">gem</span> <span class="s1">'byebug'</span>
  <span class="n">gem</span> <span class="s1">'web-console'</span><span class="p">,</span> <span class="s1">'~> 2.0'</span>
  <span class="n">gem</span> <span class="s1">'spring'</span>
<span class="k">end</span>

De la même manière, on pourrait définir un group :production pour y placer les gems qui ne doivent être incluses qu’en environnement de production.

Trouvez des gems

Comme je l’ai précisé en début d’article, le site RubyGems héberge et indexe des milliers de gems. Cependant, lorsque j’effectue une recherche, je préfère utiliser le site Ruby Toolbox.

Outre l’outil de recherche, Ruby Toolbox trie les gems par catégorie et par popularité. Cette information est souvent un bon indicateur lorsqu’on se lance dans un développement qui nécessite l’utilisation de gems inconnues. Le taux d’utilisation et la fréquence de mise à jour sont souvent déterminante dans mon choix.

Pour reprendre Devise comme exemple, si on consulte la page RubyGems consacrée à Devise, on y trouve les informations suivantes :

Les informations, reprises sur les pages RubyGems et GitHub du projet, permettent de se faire une bonne idée de la qualité du projet :

  • Le développement est actif (released about a month ago), ce qui indique probablement que la gem est compatible avec la dernière version de Rails
  • Le développement est soutenu (129 releases en 5 ans)
  • La gem est particulièrement ancienne (première release il y a 5 ans), ce qui indique probablement qu’elle est testée et à toute épreuve
  • Les related projects de la catégorie rails authentication sont tous moins bien côté, ce qui signifie que Devise est la gem la plus populaire dans sa catégorie

S’assurer de la qualité {#sassurer-de-la-qualité}

Pour s’assurer de la qualité d’un code, le mieux serait de le parcourir entièrement. Cette idée est bien entendu irréalisable au quotidien, nous allons donc essayer d’automatiser les choses pour se donner une vue d’ensemble.

Nous allons télécharger le code de Devise depuis son dépôt GitHub :

$ cd /tmp
$ git clone https://github.com/plataformatec/devise.git
$ cd devise/

Grâce à cloc, nous allons analyser le contenu des dossiers app et lib (là ou le code se situe) :

$ cloc app lib
     101 text files.
      98 unique files.
      25 files ignored.
